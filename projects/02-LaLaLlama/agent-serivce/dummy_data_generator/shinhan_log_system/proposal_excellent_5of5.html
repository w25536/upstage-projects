4. 프로젝트 수행 전략

4.1 기술 아키텍처

  • 논리/물리 아키텍처
  • 논리: UI Layer → API Gateway → 인증/권한 관리 서비스 → 마이크로서비스(로그 조회, 권한 관리, 감사 추적, 데이터 처리) → 데이터레이크/데이터베이스 → 실시간 분석 파이프라인(Kafka, Kafka Streams/ksqlDB) → 시각화 대시보드
  • 물리: 프런트는 CDN + SSR 가능 React 18 앱, 백엔드는 Kubernetes 상의 컨테이너화된 마이크로서비스, 데이터 저장은 PostgreSQL + Read Replica + 데이터 Lake(S3/HDFS) + Elasticsearch
  • 기술 스택 및 선택 근거(벤치마크 포함)
  • Spring Boot 2.7: 금융권 검증 케이스 다수, 보안 패치 속도가 신속하고 JPA 성능 우수
  • React 18: Virtual DOM 및 컴포넌트 재사용성, 금융권 사례에서 개발 생산성 증가
  • PostgreSQL: ACID 보장, 복잡 쿼리 최적화에 강점, 금융권 표준 도입 사례 다수
  • Kafka 기반 실시간 처리: 저지연 이벤트 스트리밍, 로그 이벤트의 안정적 전달
  • 데이터 시각화: D3.js/Chart.js/ApexCharts를 통한 대시보드 구성
  • 마이크로서비스 설계 전략
  • 서비스 분리 기준: 로그 조회/권한 관리/감사 추적/데이터 처리/메타데이터 관리
  • API 게이트웨이: 인증·권한 부여, 라우팅, 속도 제한
  • 서비스 메시: 안정적 호출, 트래픽 관리, 분산 트레이싱
  • 데이터 모델링 전략
  • ERD 설계: 로그 엔티티, 메타데이터 엔티티, 권한 엔티티, 감사 로그 엔티티
  • 파티셔닝 전략: 로그 테이블은 수평 파티셔닝, 메타데이터는 보존 기간별 아카이브
  • 데이터 품질 및 정합성 관리: 데이터 품질 규칙 엔진, 자동 리컨실리에이션 파이프라인

4.2 핵심 기능 설계

  • 로그 조회 UI/UX
  • 와이어프레임: 검색, 필터, 메타데이터 기반 쿼리 생성, 상세 로그 화면, 감사 로그 보기
  • 사용자 시나리오: 일반 직무/IT/컴플라이언스 직무별 대시보드 공유, 권한별 정보 가시성
  • 권한/승인 관리 프로세스
  • 프로세스 흐름: 요청 → 검토 → 승인 → 만료
  • 상태 다이어그램: 요청(대기) → 검토 중 → 승인/반려 → 만료/종료
  • 민감정보 제어 및 마스킹
  • 정책 엔진: 데이터 필드 마스킹 규칙, 노출 레벨(Role/ABAC 기반) 설정
  • 민감정보 관리: 승인된 사용자만 민감정보 조회 가능
  • 감사 추적 시스템
  • 로그 스키마: 조회/수정/생성 이벤트에 대한 상세 필드 포함
  • 보존 정책: 3년 보관, 위변조 방지 해시, 무결성 검사 주기적 수행
